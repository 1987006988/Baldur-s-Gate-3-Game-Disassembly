# Implementation Validation

## Question

在不依赖私有开发资料的前提下，我们如何尽可能验证前述体验与系统判断不是空想？

## Short answer

本模块的职责不是反向工程出完整实现，而是建立一套“公开可验证证据链”。我们优先使用官方术语、开发者访谈、补丁说明、UI 信号、流程结构、数据组织线索与稳定可复述现象，来判断前述分析哪些已足够稳，哪些仍只是合理猜测。

## 验证入口

- 官方资料中的设计意图与术语
- 补丁说明中的系统修正点
- UI、日志、提示、状态变化中的稳定信号
- 模块之间是否存在一致的回流现象
- 高辨识度案例能否在补丁 / 热修 / 社区更新中找到明确的状态差异与后续反馈锚点

## 当前工作假设

- `推断`：如果一个体验在多个模块反复出现，且官方用相近术语描述，就可以提高其可信度。
- `推断`：补丁说明能暴露设计团队最在意的系统边界和失效点。
- `推断`：当官方补丁明确区分某个分支的状态差异，并同时修复其跨长休、跨区域或后续章节反应时，这类条目可作为“状态回流存在”的强验证入口。
- `待验证问题`：是否能从公开工具链、脚本命名或社区技术观察中提取更具体的实现线索。

## 证据使用原则

- 没有来源支撑的技术判断不升级为事实
- 观察到的 UI 或流程信号，要和来源材料互相校对
- 发现冲突时优先降级结论，而不是硬拼解释
- 若某个案例只有社区线索而缺乏官方直接来源，只能把它留在“案例入口”层，不能直接升级为系统性事实

## 与当前任务模块对齐的验证入口

- `Minthara 击倒路径`：
  - `BG3-OFF-004` 直接暴露了 `knocked out` / `killed` 的状态区分，以及跨长休判定失效点。
  - `BG3-OFF-002` 直接暴露了该状态在后续区域与后续章节中的再调用。
- `Dark Urge 营地 bard 事件`：
  - `BG3-OFF-005` 直接暴露了“营地临时入队”这一状态变化，但仍未暴露完整后果分布，因此目前只能作为辅验证入口。

这两条案例共同说明：补丁、热修与社区更新不是边缘材料，而是当前最稳定的公开实现验证锚点之一。
