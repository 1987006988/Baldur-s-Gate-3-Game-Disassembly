# Master Roadmap

## 作用

本文件是《博德之门3》研究仓库的项目级主路线图。它不替代 ExecPlan，也不展开到逐条任务；它的职责是为未来每一轮 Codex 工作提供稳定主线，回答三个问题：

1. 现在整个项目在往哪里走。
2. 下一阶段该优先推进什么，为什么。
3. 什么算完成，什么算跑偏。

## 项目总目标

用三层拆解方法，建立一套可持续修正、可追溯到来源、可被后续会话接力的 BG3 研究框架，最终回答这个总命题：

> 《博德之门3》如何把桌面规则、电影化叙事和系统性玩家能动性缝合成一个高反应性的 CRPG。

## 非目标

- 不做全剧情整理或章节复述。
- 不做全量百科式资料搬运。
- 不为“看起来完整”而把每个子系统拆成孤立条目。
- 不在缺乏来源时提前下技术实现结论。
- 不把聊天上下文当作长期记忆。

## 项目模块

### 模块 1：项目记忆与治理

职责：维护仓库入口、状态、决策、路线图和执行规则，确保新会话能快速恢复上下文。

完成标准：

- `README.md`、`AGENTS.md`、`docs/AGENTS.md`、`current_state.md`、`next_step.md`、`decision_log.md`、`master_roadmap.md` 互相一致。
- 新会话只读项目入口文件即可知道目标、阶段、主任务和协作规则。
- 任何阶段切换都能在状态文件中看出来。

### 模块 2：来源体系与证据管线

职责：把来源从“待收集”变成“可引用、可追踪、可复核”的证据库。

完成标准：

- `source_index.md` 中的核心来源有稳定分组、状态和 `Where used`。
- 来源笔记模板被实际复用，至少形成首批可复核来源摘要。
- 各类来源的优先级和用途边界被稳定执行，而不是停留在规则层。

### 模块 3：总论点与宏观结构

职责：先建立全项目的解释框架，再看局部模块，避免后续分析只剩局部案例堆叠。

对应文档：

- `docs/03_analysis/00_core_thesis.md`
- `docs/03_analysis/01_macro_structure.md`

完成标准：

- 能清楚回答 BG3 的“高反应性”总论点当前版本是什么。
- 宏观结构能解释 Act、区域、入口、推进方式为何服务于自由感。
- 总论点与宏观结构都有明确的来源支撑和待验证问题。

### 模块 4：任务与选择回流

职责：解释 BG3 的高反应性为何不只是分支数量，而是状态回流、延迟反馈和多入口解法。

对应文档：

- `docs/03_analysis/02_quests_and_choices.md`

完成标准：

- 至少建立一条“选择点 - 状态变化 - 后续反馈”的清晰证据链。
- 至少有 2 个可复述案例，且能区分事实、推断、待验证问题。
- 能说清它与宏观结构、营地循环、战斗解法的耦合点。

### 模块 5：同伴、营地与长休循环

职责：解释叙事陪伴感为何能被系统节奏放大，而不是只靠角色塑造成立。

对应文档：

- `docs/03_analysis/03_party_and_camp.md`

完成标准：

- 能说明营地/长休为什么是反馈放大器，而不只是休整界面。
- 至少有 1 到 2 条“任务或战斗结果 - 营地/同伴反馈”的证据链。
- 能指出它与任务推进、资源恢复、关系变化的交汇机制。

### 模块 6：战斗、构筑与环境战术

职责：解释玩家为什么觉得“我想到的办法能成立”，以及这种感觉如何来自规则与空间的共同设计。

对应文档：

- `docs/03_analysis/04_combat_and_environment.md`

完成标准：

- 至少能区分“职业/构筑价值”“环境交互价值”“战前布局价值”三类贡献。
- 至少有 2 个案例能说明环境与规则是如何共同扩展解法空间的。
- 能指出哪些判断是通用系统，哪些更像个别场景演出。

### 模块 7：实现验证与全局综合

职责：把前面模块的判断放回公开可验证证据里，形成全项目可收束的解释框架。

对应文档：

- `docs/03_analysis/05_implementation_validation.md`
- `docs/03_analysis/00_core_thesis.md`

完成标准：

- 每个核心模块都有至少一组实现验证入口，不再只停留在体验描述。
- 能把官方术语、开发者说明、UI 信号、补丁说明与模块判断对应起来。
- 总论点完成一次基于证据的回写与收束。

## 推荐执行顺序

推荐顺序：

1. 模块 1：项目记忆与治理
2. 模块 2：来源体系与证据管线
3. 模块 3：总论点与宏观结构
4. 模块 4：任务与选择回流
5. 模块 5：同伴、营地与长休循环
6. 模块 6：战斗、构筑与环境战术
7. 模块 7：实现验证与全局综合

顺序理由：

- 先做模块 1，是因为没有稳定记忆和状态机制，后续所有研究都难以接力。
- 再做模块 2，是因为没有证据管线，分析会很快滑成印象判断。
- 模块 3 先于具体系统模块，是为了先建立全局解释框架，避免后续每个模块各说各话。
- 模块 4 优先于 5 和 6，因为“任务与选择回流”最直接承载 BG3 的高反应性命题，也是目前仓库已明确的首个深挖对象。
- 模块 5 紧随其后，因为营地和长休经常是任务反馈的二次显现层。
- 模块 6 放在其后，是因为战斗与环境需要在前述主线已稳定后，再判断它们是独立亮点还是共同支撑反应性的组成部分。
- 模块 7 最后集中收束，因为它的价值在于校正和整合前面模块，而不是脱离上下文单独展开。

## 阶段划分与出口条件

### 阶段 A：基础就绪

范围：模块 1 完成，模块 2 起步。

出口条件：

- 项目入口、状态文件、路线图和规则已稳定。
- 新会话可在 10 分钟内恢复上下文并找到当前主任务。
- 仓库检查命令通过。

### 阶段 B：证据底座建立

范围：模块 2 完成，模块 3 起步。

出口条件：

- 已有一批可直接引用的核心来源。
- `source_index.md` 不再只是占位索引，而开始承担证据导航作用。
- 至少一个核心模块具备可操作的来源入口。

### 阶段 C：核心模块穿透

范围：模块 3、4、5、6 逐步完成。

出口条件：

- 总论点、宏观结构、任务与选择、营地循环、战斗环境都各自完成首轮证据链。
- 每个核心模块都能明确区分事实、推断、待验证问题。
- 模块之间的耦合关系已被写清，而不是孤立分析。

### 阶段 D：综合校正与收束

范围：模块 7 完成。

出口条件：

- 关键命题都能映射到来源与验证入口。
- 已形成一版可供后续长期迭代的“当前最强解释”。
- `next_step.md` 不再是补骨架，而是指向局部深化、纠错或扩展。

## 什么算跑偏

出现以下任一情况，应视为跑偏并及时纠正：

- 大量新增文字，但没有来源 ID、没有待验证问题、没有状态同步。
- 把剧情复述、角色介绍或系统罗列误当成拆解分析。
- 只堆来源，不回写到模块；或只写模块，不补来源索引。
- 在没有证据时直接声称具体实现细节已经确定。
- 新增结构明显增加维护成本，却没有提升上下文恢复速度或证据可追溯性。
- 一轮工作结束后，`current_state.md` 和 `next_step.md` 仍不能告诉下一会话该做什么。

## 使用方式

- 每次启动新一轮工作前，先用本文件判断当前处于哪个阶段、应该推进哪个模块。
- 遇到大型模块推进、方法变化或结构重排时，先建立 ExecPlan，再动正文。
- 路线图只在项目主线改变时更新；局部任务变化应优先写进 `current_state.md` 和 `next_step.md`。
