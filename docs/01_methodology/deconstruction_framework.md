# Deconstruction Framework

## 核心问题

我们不是只问“BG3 有哪些内容”，而是问“为什么玩家会把这些内容体验成高自由度、高沉浸感和高反馈性”。

## 玩家体验层看什么

- 玩家在什么节点感到可选择而非被推着走
- 玩家如何收到来自世界、同伴、任务或战斗结果的反馈
- 玩家为什么觉得自己的构筑、路径和处置方式被认可
- 哪些体验来自叙事包装，哪些来自系统深度，哪些来自二者缝合

## 规则系统层看什么

- 任务与选择如何记录状态并在后续回流
- 长休、营地、同伴关系、资源消耗如何互相牵引
- 战斗、探索、成长如何共享资源、风险和解法空间
- 区域结构、入口设计与内容推进如何支持非线性体验

## 实现验证层看什么

- 官方公开资料中的设计表述与术语
- 开发者访谈、补丁说明、社区更新中的系统信号
- UI 呈现、日志、提示、流程结构中可观察到的实现线索
- 可被验证的数据组织、脚本命名、工具链与工作流痕迹

## 三层如何互相校正

- 体验层提出要解释的现象
- 系统层给出机制解释
- 实现层检验这些机制解释是否有证据支撑
- 如果实现层证据不足，就把结论降级为推断或待验证问题

## 如何避免滑向剧情复述或功能罗列

- 不只记“发生了什么”，而要记“为什么这一安排会增强自由感或反馈感”
- 不只写“系统存在”，而要写“它和哪些系统耦合、制造了什么行为结果”
- 每个模块至少保留一个实现验证问题，逼迫后续继续找证据
